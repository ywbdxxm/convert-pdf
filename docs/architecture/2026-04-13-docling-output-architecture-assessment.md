# Docling Output Architecture Assessment

Date: 2026-04-13

## Question

For the current `manuals/processed/<doc_id>` structure:

1. Is this the optimal output architecture for Docling?
2. Which artifacts are generated by Docling natively?
3. Which artifacts are added by our own wrapper logic?

## Short Answer

No, the current `manuals/processed/<doc_id>` layout is not proven to be the universal best architecture for Docling.

It is better described as:

- a strong, agent-friendly bundle for our current use case
- a customized Docling packaging layer
- not a canonical Docling standard

For Docling specifically, the better default is likely:

```text
Docling native outputs
-> keep them intact
-> add only a thin overlay for navigation / quality / comparison
```

not:

```text
mix native outputs and custom derived artifacts into one undifferentiated schema
```

## Current Structure

Today a processed document directory typically looks like:

```text
manuals/processed/<doc_id>/
  document.json
  document.md
  document.html
  manifest.json
  sections.jsonl
  chunks.jsonl
  alerts.json
  artifacts/
  tables/
  _windows/
```

This structure is useful, but it blends together:

- Docling-native exports
- Docling-native APIs used by our wrapper
- custom schemas
- custom heuristics
- custom caching

That blending is why it is practical, but not cleanly optimal.

## What Is Native Docling

These outputs or capabilities come directly from Docling / docling-core APIs.

### Direct native exports

- `document.json`
  - from `DoclingDocument.save_as_json(...)`
- `document.md`
  - from `DoclingDocument.save_as_markdown(...)`
- `document.html`
  - from `DoclingDocument.save_as_html(...)`
- `artifacts/`
  - image sidecars generated when Docling exports referenced images

### Native structural APIs we use

- `DocumentConverter`
- `ThreadedStandardPdfPipeline`
- `HybridChunker`
- `DoclingDocument.iterate_items()`
- `TableItem.export_to_dataframe(...)`
- `TableItem.export_to_html(...)`
- `TableItem.export_to_markdown(...)`

### Native but wrapped by us

These are based on Docling-native capabilities, but their existence in the current folder is our choice:

- `tables/*.csv`
  - table content comes from native table export APIs
  - directory layout, naming, and exporting every table are our decisions
- `tables/*.html`
  - same as above

## What Is Custom Packaging

These artifacts are not Docling-native standards. They are ours.

### Purely custom files

- `manifest.json`
- `sections.jsonl`
- `chunks.jsonl`
- `alerts.json`
- `_windows/`
- `manuals/processed/_runs/<timestamp>.json`

### Purely custom behaviors

- `doc_id` naming rules
- output directory layout
- window cache/resume behavior
- SHA-256 recording
- run summary generation
- table sidecar manifest records
- related-table overlap attachment into chunk/section records
- explicit alert semantics

### Custom heuristics layered on top of native data

- caption recovery
- Markdown table-sidecar link injection
- noisy chunk filtering
- image filtering heuristics
- image-fallback table detection
- empty table sidecar alert detection

## Practical Native vs Custom Map

| Artifact / behavior | Native Docling | Custom packaging |
| --- | --- | --- |
| `document.json` | Yes | No |
| `document.md` | Yes | We also post-process it |
| `document.html` | Yes | No |
| `artifacts/` | Yes | Path/layout chosen by us |
| `tables/*.csv` | Content from native table APIs | Export policy, naming, layout are ours |
| `tables/*.html` | Content from native table APIs | Export policy, naming, layout are ours |
| `manifest.json` | No | Yes |
| `chunks.jsonl` | Built from native `HybridChunker` | Schema, filtering, fields are ours |
| `sections.jsonl` | No | Yes |
| `alerts.json` | No | Yes |
| `_windows/` cache | No | Yes |
| `_runs/` summaries | No | Yes |
| table links inside Markdown | No | Yes |
| caption recovery | No | Yes |
| page/image heuristics | No | Yes |

## Is The Current Structure Optimal For Docling?

Not fully.

It is strong in these ways:

- convenient for an agent
- easy for a human to inspect
- preserves both readable and machine-readable forms
- exposes some quality issues explicitly
- already works on real manuals

But it is not optimal because:

- it mixes native and custom layers together
- it hides where Docling stops and our wrapper begins
- it makes parser-vs-packaging comparisons harder
- it encourages schema lock-in

## Better Docling-Oriented Architecture

If we were designing the Docling path more cleanly today, a better structure would probably be:

```text
<doc_root>/
  raw/
    document.json
    document.md
    document.html
    artifacts/
    tables/
  overlay/
    manifest.json
    index.json
    quality-summary.md
    pages/
    chunks.filtered.jsonl
    chunks.native.jsonl
    sections.jsonl
    alerts.json
```

Why this is better:

- `raw/` preserves what Docling actually produced
- `overlay/` makes our packaging explicit
- comparisons against OpenDataLoader become cleaner
- heuristics become optional rather than hidden defaults

## Practical Conclusion

For the current project:

- the existing `manuals/processed/<doc_id>` layout is useful and good enough as a baseline
- it is not the final word on best practice
- its strongest ideas are:
  - one document directory
  - readable + machine-readable outputs
  - explicit risk/provenance files
- its weakest property is:
  - mixing native Docling outputs and custom derived artifacts into one implicit schema

So the answer is:

> `manuals/processed/<doc_id>` is a good current bundle for our Docling path, but not a proven universal optimum.

And:

> the cleanest future refinement is to separate native Docling outputs from our overlay logic, not to pile on more heuristics.
