# 四份产物详细对比分析

日期：2026-04-15

## 1. 目标与评价标准

本分析只看一件事：

> 对 `Codex`、`Claude Code` 这类 code agent 来说，这四份最终产物到底好不好用。

这里的“好用”不看 parser 名气，不看 schema 美感，而看：

1. 入口是否明确
2. 导航是否清晰
3. 文本是否适合直接阅读
4. 表格是否容易核验
5. 图片证据是否完整
6. 风险页是否容易暴露
7. 是否容易回到原 PDF
8. 目录噪音是否足够低

本次对比对象只有四份：

1. `opendataloader_hybrid / ESP32-S3 datasheet`
2. `opendataloader_hybrid / ESP32-S3 TRM`
3. `docling_bundle / ESP32-S3 datasheet`
4. `docling_bundle / ESP32-S3 TRM`

## 3. 各方案的真实优缺点

### 3.1 OpenDataLoader Hybrid 的优点

1. 证据层最强。
   - `elements.index.jsonl` 直接保留 element 级信息、页码和 bbox，对 forensic 检查非常强。

2. 表格覆盖率明显更高。
   - 在 ESP32-S3 TRM 上，结构化表格数量显著高于 `docling_bundle`。

3. 体积控制更好。
   - 最新结构收敛后，datasheet 只有 `5.2M`，TRM 只有 `53M`。

4. 最终目录已经足够干净。
   - 单入口、无 `pages/`、无 `runtime/native`、无 staging 泄露。

### 3.2 OpenDataLoader Hybrid 的缺点

1. 直接阅读体验仍然不够 calm。
   - `document.md` 行数更长，TRM 达到 `55525` 行。
   - 虽然证据强，但长时间直接阅读不如 Docling 顺手。

2. 当前导航层偏“证据导向”，不够“阅读导向”。
   - 只有 `elements.index.jsonl`，对 agent 做精确证据定位很好。
   - 但如果任务是“我想先按主题找一段上下文再展开”，它不如 `sections/chunks` 自然。

3. 大 TRM 依赖 fallback。
   - TRM 的 `fallback_detected = true`
   - 说明 ODL 作为 parser 本体还不是无条件稳定，仍依赖兜底路径。

4. 对 hard table 的告警能力仍然偏窄。
   - datasheet 的 page 27 能抓到
   - 但当前告警模型主要盯 “heading + image + no native table”
   - 对更多退化模式还不够全面

### 3.3 Docling Bundle 的优点

1. 阅读工作流更成熟。
   - `document.md` 更 calm
   - `sections.jsonl + chunks.jsonl` 更适合“先按主题定位，再看页级片段”

2. 入口文件更适合日常工程使用。
   - `README.md` 结合 `alerts.json`，对 agent 的起手动作更自然。

3. 对“阅读 + 逐步核验”的路径更友好。
   - 不是纯 forensic，而是更像工程师平时查手册的流程。

4. 风险暴露更保守。
   - 在 TRM 上，`9` 个 alerts 明确暴露了空 sidecar 等问题。
   - 虽然不是优点中的“漂亮结果”，但从工程安全角度它比沉默更好。

### 3.4 Docling Bundle 的缺点

1. 结构化表格覆盖率明显不如 ODL。
   - TRM：`667` vs `2467`
   - 这会直接影响 agent 查大量表格型信息时的效率。

2. 最终体积明显更大。
   - datasheet: `16M`
   - TRM: `203M`
   - 主要不是表格，而是 `assets/` 和主文件更重。

3. 当前导航层虽然更适合阅读，但也更“重”。
   - 同时保留 `sections.jsonl` 与 `chunks.jsonl`
   - 对某些任务来说是优势
   - 但从长期统一 agent 接口看，仍可能过于分叉

4. 告警暴露了实实在在的 parser 边界。
   - 比如 TRM 中多处 `empty_table_sidecar`
   - 这说明 Docling 的某些 figure-like / table-like 页面依然不能稳妥结构化

## 4. 四份产物逐项对比

### 4.1 Datasheet 对比

#### OpenDataLoader Hybrid / Datasheet

优点：

- 小
- 结构化证据强
- page 27 的 hard-table 告警抓得准
- element 索引非常适合精确检查

缺点：

- 主阅读副本偏吵
- 对“顺着主题读”不够自然

#### Docling Bundle / Datasheet

优点：

- 主阅读副本更顺手
- 章节与 chunk 导航更贴近日常阅读
- 告警表达更贴近“文档阅读问题”

缺点：

- 更大
- 图像资产更重
- 结构化证据层不如 ODL 精细

#### 当前 datasheet 结论

如果当前任务是：

- 精确核证据、看 bbox、查复杂页面
  - 选 `OpenDataLoader hybrid`

- 快速日常查阅、理解上下文、再回表格核验
  - 选 `docling_bundle`

### 4.2 TRM 对比

#### OpenDataLoader Hybrid / TRM

优点：

- 结构化表格最强
- element 数量和细粒度证据最强
- 体积明显更可控

缺点：

- runtime 上依然暴露出 fallback 依赖
- 阅读工作流仍不如 Docling calm
- 对 agent 来说更像“证据机器”，不像“手册阅读器”

#### Docling Bundle / TRM

优点：

- 阅读与章节导航更自然
- 风险暴露更明确

缺点：

- 体积极大：`203M`
- 表格覆盖率远弱于 ODL
- `9` 个 alerts 说明有不少边界页必须额外小心

#### 当前 TRM 结论

TRM 这种超大文档上：

- `OpenDataLoader hybrid` 是更强的结构化证据方案
- `docling_bundle` 是更强的阅读/上下文方案
- 但没有任何一条线可以完全替代另一条线

## 5. 具体哪里做得还不够好

### 5.1 两条线共同的问题

1. 导航层仍未统一。
   - ODL 用 `elements.index.jsonl`
   - Docling 用 `sections + chunks`
   - 对最终 agent 接口来说，这仍是分叉

2. `document.html` 是否默认保留，还没有被彻底证明。
   - 它有价值
   - 但默认总保留是否最佳，还没有经过更细的任务验证

3. `manifest.json` 仍然只是“比较干净”，不是“最终极简”。
   - 已经比之前好很多
   - 但未来还可以继续收敛

### 5.2 OpenDataLoader 具体问题

1. 直接阅读体验仍偏硬。
2. 大 TRM 依然依赖 fallback。
3. 告警模型还不够宽。
4. 当前没有一个更“阅读导向”的统一 locator 层。

### 5.3 Docling 具体问题

1. 大文档体积过大。
2. 结构化表格覆盖率明显偏低。
3. `empty_table_sidecar` 暴露出稳定边界问题。
4. `sections + chunks` 虽然好用，但从长期统一接口看仍可继续收敛。

## 6. 是否还有值得做的优化

有，而且不是边角级别。

### 6.1 值得继续做的

1. 统一导航层。
   - 继续评估是否引入统一 `locator.index.jsonl`
   - 让两条线最终对 agent 的导航接口更接近

2. 继续扩展 ODL 的风险告警。
   - 不只盯一种 hard-table 退化模式

3. 继续审查 Docling 的空表格 sidecar 页面。
   - 对这类页面增加更明确的风险标签

4. 继续清理过时文档。
   - 当前还有一批长文档仍在讲旧结构

### 6.2 不该做的

1. 不该再把 `pages/` 加回来当默认层。
2. 不该再把 `runtime/native/cache` 混回最终 bundle。
3. 不该因为体积大就去 heuristic 删图片。
4. 不该在没有必要的情况下重新扩 parser 面。

## 7. 当前阶段的管理判断

当前阶段最准确的管理判断是：

> 项目已经从“实验性工程”进入“可正式使用的基础设施”阶段，但还没有彻底定型。

更具体地说：

- 已经明显比之前更接近最佳实践
- 但仍存在清楚的后续优化空间
- 这些优化空间不是“锦上添花”，而是围绕统一导航、风险表达、长期维护成本的继续收敛

## 8. 当前总评

如果按当前四份产物给一个务实结论：

- 最强证据提取器：`OpenDataLoader hybrid`
- 最强阅读 bundle：`docling_bundle`

如果从“整体给 code agent 用”的当前总体验看：

- `docling_bundle` 仍略占优势

原因不是它解析更强，而是：

- 更 calm
- 更自解释
- 更适合日常“先读、再查、再核验”

但如果任务偏向：

- 复杂表格
- forensic 证据核查
- 结构化页级元素分析

那么 `OpenDataLoader hybrid` 的优势会更明显。
