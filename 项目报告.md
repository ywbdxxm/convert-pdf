# AI 时代嵌入式开发手册基础设施项目报告

日期：2026-04-15

## 一、项目定义

本项目不是一个通用 PDF 转 Markdown 小工具，也不是一个先做聊天 UI 或 RAG 应用的外围系统。

它的真实定位是：

> 为 AI 时代的嵌入式开发，构建一套把芯片数据手册、技术参考手册、应用笔记等 PDF 转换为可供 agent 直接使用的结构化参考资产的基础设施。

这里的“agent”主要指：

- Codex
- Claude Code
- 其他以代码与终端为主要工作环境的 AI 编码代理

项目的核心目标不是“把 PDF 转出来”，而是让 agent 在真实工程中具备下列能力：

- 快速定位寄存器、引脚、时序、电气参数等关键信息
- 保留证据链，能回到章节、表格、图、页码
- 知道哪些页面不应盲信，必须回原 PDF 核验
- 在日常嵌入式开发中长期稳定复用

## 二、需求背景

在嵌入式开发场景里，AI 参与工作的真正瓶颈，不是“能不能看 PDF”，而是：

- 面对几百页到几千页手册时，能不能足够快地找到正确位置
- 面对大量表格、寄存器摘要、框图、时序图时，能不能保留证据完整性
- 面对 parser 的误判和退化页时，能不能显式暴露风险
- 面对真实工程问题时，能不能从“查到结果”走到“给出可核验结论”

因此，本项目从一开始就不把目标设定为：

- “转成 Markdown 就算成功”
- “先搭一个向量库再说”
- “先做一个聊天界面”

而是直接从第一性原理出发：

1. 原始 PDF 必须始终保留为最终权威
2. 结构化产物的价值在于加速定位、降低查阅成本、暴露证据与风险
3. 最终评价标准必须是 agent 的真实使用效果，而不是 parser 自身的学术纯度

## 三、当前总体方案

当前方案保留两条正式产线：

1. `docling_bundle`
2. `opendataloader_hybrid`

对应的原始输入与最终输出如下：

- 原始 PDF：
  - `manuals/raw/`
- 最终 bundle：
  - `manuals/processed/docling_bundle/<doc_id>/`
  - `manuals/processed/opendataloader_hybrid/<doc_id>/`
- staging / cache：
  - `tmp/opendataloader_hybrid-*-native/`
  - `tmp/docling_bundle-cache/`

这两条产线不是“二选一”的关系，而是当前阶段的两条互补路线：

- `OpenDataLoader hybrid` 更强于结构化证据提取
- `docling_bundle` 更强于阅读与验证工作流

## 四、设计原则

### 4.1 原始 PDF 始终是最终权威

无论任何解析器输出多好，下列内容都不能脱离原 PDF 直接盲信：

- 寄存器位定义
- pin 映射
- 时序限制
- 电气特性
- 图形化时序图、框图、宽表

因此，解析结果的定位是：

- 加速定位
- 降低查阅成本
- 保留证据
- 提示风险

而不是替代原始手册。

### 4.2 以 agent 使用效果为最高评价标准

当前真正的评价标准是：

- 入口是否明确
- 导航是否清晰
- 表格和图片是否容易核验
- 风险页是否容易暴露
- 是否容易回到原 PDF

不再以这些标准作为第一目标：

- parser 是否“看起来更纯”
- schema 是否“形式上更统一”
- 输出目录是否“为了统一而统一”

### 4.3 保留完整图片证据

当前已经明确采用：

- `assets/` 默认完整保留
- 不使用 heuristic 去删除图片

原因是：

- 时序图、框图、寄存器图、视觉化表格经常是后续真正要用的证据
- 图片误删带来的损失，高于 bundle 体积偏大的代价

### 4.4 最终 bundle 只保留 agent 真正会用的层

当前默认保留的是：

- 单入口 `README.md`
- `manifest.json`
- `alerts.json`
- `document.md`
- `document.json`
- `document.html`
- 一套导航层
- `tables.index.jsonl`
- `tables/*.csv`
- `assets/`

运行期 staging、native dump、cache 统一留在 `tmp/`，不再进入最终 bundle。

## 五、架构设计

### 5.1 工作站分层

当前项目使用的是分层工作站架构：

1. WSL 系统层
2. 共享 AI base
3. 项目 overlay 环境
4. 原始 PDF / 最终 bundle / staging 数据

实际结构为：

- 系统层：
  - Java 17
  - Docker
  - NVIDIA runtime
  - OCR 与构建工具
- 共享 AI base：
  - `/home/qcgg/.mamba/envs/ai-base-cu124-stable`
- 项目 overlay：
  - `docling/.venv`
  - `opendataloader/.venv`

这个分层的目标是避免每个工具重复安装大体积 GPU 依赖。

### 5.2 代码层架构

当前核心代码分为两套：

- `docling_bundle/`
- `opendataloader_hybrid/`

这两者都不是 parser 本身，而是：

> 面向 agent 的最终 bundle 组织器。

### 5.3 数据流架构

当前更准确的数据流是：

```text
PDF -> parser -> agent-facing bundle -> agent 使用 -> 原始 PDF 核验
```

不再是旧思路里那种：

```text
PDF -> parser -> 尽可能多的派生文件 -> 让 agent 自己猜该看哪层
```

## 六、当前两种方案

### 6.1 方案一：Docling Bundle

#### 设计目标

`docling_bundle` 的目标不是“证明 Docling 原生输出最纯”，而是：

> 把 Docling 的结构能力与阅读能力组织成一套更适合 agent 日常查阅与验证的 bundle。

#### 当前默认 bundle 结构

```text
<doc_id>/
  README.md
  manifest.json
  alerts.json
  document.md
  document.json
  document.html
  sections.jsonl
  chunks.jsonl
  tables.index.jsonl
  tables/
  assets/
```

#### 当前优势

- 阅读体验更 calm
- 章节导航更自然
- 更适合日常“先读、再查、再核验”
- table sidecar workflow 更顺手

#### 当前短板

- bbox / spatial metadata 明显弱于 ODL
- 结构化表格覆盖率低于 ODL
- figure-like table 与 empty sidecar 仍是边界问题
- 大型 TRM 更重、更吃资源

### 6.2 方案二：OpenDataLoader Hybrid

#### 设计目标

`opendataloader_hybrid` 的目标不是把 ODL 改造成 Docling，而是：

> 保留 ODL 强结构证据层的优势，再把它收成 agent 能直接用的 bundle。

#### 当前默认 bundle 结构

```text
<doc_id>/
  README.md
  manifest.json
  alerts.json
  document.md
  document.json
  document.html
  elements.index.jsonl
  tables.index.jsonl
  tables/
  assets/
```

#### 当前优势

- bbox / page metadata 更强
- 结构化表格覆盖率明显更高
- 对复杂页面的证据核查能力更强
- 提取速度更快
- GPU 压力更不持续

#### 当前短板

- 大 TRM 仍依赖 `--hybrid-fallback`
- 直接阅读体验仍不如 Docling calm
- 某些 hard table 仍会退化成 `image + text fragments`

## 七、代码实现要点

### 7.1 已完成的重要结构收敛

这一轮已经落地的关键改动包括：

- 两条产线统一为单入口 `README.md`
- 统一视觉资产目录为 `assets/`
- 默认仅保留 CSV table sidecar
- `docling_bundle` 的 window cache 移到 `tmp/docling_bundle-cache/`
- `opendataloader_hybrid` 的运行状态折叠进 `README.md` / `manifest.json`

### 7.2 为什么这些改动重要

这些改动的意义不是“让目录更干净”，而是：

- 减少 agent 决策成本
- 避免多入口文件并存造成的分叉
- 避免 debug 层污染最终使用路径
- 让阅读、定位、核验的默认流程更稳定

## 九、当前 bundle 大小与现实判断

最新一轮 clean rerun 后，四份 bundle 体积如下：

- `opendataloader_hybrid` datasheet: `5.2M`
- `opendataloader_hybrid` TRM: `53M`
- `docling_bundle` datasheet: `16M`
- `docling_bundle` TRM: `203M`

这说明：

- 结构层冗余已经大幅收掉
- 当前主要体积来源已经不在结构冗余层
- 而是 `assets/` 和主文件本身

因为我们已经决定完整保留图片证据，所以当前不把“继续删图片”作为默认优化方向。

## 十、当前结论

如果按“提取证据能力”评判：

- `OpenDataLoader hybrid` 更强

如果按“给 Codex / Claude Code 直接查阅和验证的工作流”评判：

- `docling_bundle` 更强

因此当前最合理的管理结论不是“只留一个”，而是：

- 保留两者
- 把它们视为互补产线

如果现在必须二选一：

- 最强证据提取器：`OpenDataLoader hybrid`
- 最强阅读 bundle：`docling_bundle`

如果从“整体给 Codex 用”的当前总体验看：

- `docling_bundle` 仍略占优势

原因不是它解析更强，而是：

- 更 calm
- 更自解释
- 更适合日常“先读、再查、再核验”

## 十一、四份最终产物的对比结论

当前真正有价值的比较，不应该只停留在“方案 A vs 方案 B”的抽象层，而必须落到四份真实产物：

1. `opendataloader_hybrid / ESP32-S3 datasheet`
2. `opendataloader_hybrid / ESP32-S3 TRM`
3. `docling_bundle / ESP32-S3 datasheet`
4. `docling_bundle / ESP32-S3 TRM`

### 11.1 Datasheet 对比

`OpenDataLoader hybrid / datasheet` 的特点是：

- 体积更小：`5.2M`
- 结构化证据更强
- `elements.index.jsonl` 对页级核查很强
- page 27 的 hard-table 告警抓得准

但它的问题是：

- `document.md` 仍偏“证据导向”
- 阅读体验不如 Docling calm

`docling_bundle / datasheet` 的特点是：

- 阅读体验更顺手
- `sections + chunks` 更适合按主题先读后查
- 阅读与核验工作流更自然

但它的问题是：

- 体积更大：`16M`
- 结构化证据精细度不如 ODL

### 11.2 TRM 对比

`OpenDataLoader hybrid / TRM` 的特点是：

- 体积明显可控：`53M`
- 结构化表格最强：`2467` 张
- element 级证据最强

但它的问题是：

- `fallback_detected = true`
- 大 TRM 仍依赖 fallback 才够稳
- 阅读工作流仍偏硬

`docling_bundle / TRM` 的特点是：

- 阅读与章节导航更自然
- 风险暴露更清楚

但它的问题是：

- 体积极大：`203M`
- 结构化表格数量只有 `667`
- `9` 个 alerts 说明边界问题仍然明显

### 11.3 对比总评

当前最准确的总结是：

- `OpenDataLoader hybrid` 更像强证据提取器
- `docling_bundle` 更像强阅读 bundle

这也是为什么当前最合理的管理判断不是“立刻只留一个”，而是：

- 保留两者
- 明确各自优势
- 继续围绕最终 agent 工作流收敛接口

## 十二、当前具体还做得不够好的地方

### 12.1 两条产线共同问题

1. 导航层仍未统一。
   - ODL 用 `elements.index.jsonl`
   - Docling 用 `sections + chunks`

2. `document.html` 是否应默认保留，仍值得继续验证。

3. `manifest.json` 仍然不是“最终极简”，还有继续收敛空间。

### 12.2 OpenDataLoader 当前问题

1. 阅读体验仍偏硬。
2. 大 TRM 依然依赖 fallback。
3. 告警模型还不够宽。

### 12.3 Docling 当前问题

1. 大文档体积极大。
2. 结构化表格覆盖率明显偏低。
3. `empty_table_sidecar` 暴露了真实边界页问题。

## 十三、当前值得继续做的优化

1. 继续评估是否收敛成统一 `locator.index.jsonl`
2. 继续扩展 ODL 的风险告警覆盖面
3. 继续增强 Docling 对空 sidecar / figure-like table 的风险表达
4. 继续清理旧文档，让全部说明都和当前真实产物一致

## 十四、当前不该做的事

1. 不该把 `pages/` 加回默认层
2. 不该把运行期 staging、native dump、cache 混回最终 bundle
3. 不该因为体积大就 heuristic 删图片
4. 不该在没有必要时重新扩 parser 面

## 十五、当前成熟度判断

项目已经完成了从“能跑”到“可正式使用”的关键过渡。

更准确地说：

- 已经明显比之前更接近最佳实践
- 但还没有彻底定型
- 当前处于“可用且持续优化中”的阶段，而不是“完全无需再改”的阶段

## 十六、下一步重点

当前下一步最值得继续推进的点：

1. 继续统一两条产线的 agent-facing 使用方式
2. 继续评估是否需要把导航层进一步收敛成统一 `locator.index.jsonl`
3. 基于固定任务继续做 A/B 对比，而不是继续靠印象判断
4. 把旧文档口径彻底收敛到当前真实产物结构

## 十七、管理层结论

如果现在需要一句管理层结论，可以概括为：

> 项目已经完成了从“实验性工程”到“可正式使用基础设施”的关键过渡，当前两条产线都具备稳定实际价值；但围绕 agent 最终使用方式的 bundle 结构仍在继续收敛，项目处于“可用且持续优化中”的阶段，而不是“已经完全定型”的阶段。
